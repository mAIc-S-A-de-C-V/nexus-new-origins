1. Platform Architecture Overview

Nexus Origins is a microservices platform where every piece of enterprise data — regardless of where it lives — converges into a unified, versioned ontology. Eight services work together to make that happen:

:8001
Connector Service
Manages API credentials, fetches schemas, discovers sample data
:8002
Pipeline Service
DAG executor — SOURCE → transforms → SINK. Built-in scheduler.
:8003
Inference Service
Claude AI — semantic type detection, conflict resolution
:8004
Ontology Service
Object types, properties, links, versioned records, app registry
:8005
Event Log Service
Process mining event store — case/activity/timestamp
:8006
Audit Service
Immutable audit trail for every platform action
:8007
Schema Registry
Raw schema snapshots and versioning
:8008
Correlation Engine
Cross-source entity resolution and similarity scoring

End-to-End Data Flow

External API / App
→
Connector
→
Pipeline DAG
→
Ontology Record
→
Event Log

A Connector defines how to reach an external API (auth, endpoint, query params). A Pipeline pulls data from one or more connectors, applies transformations (filter, map, enrich, dedupe), and pushes the result to an Object Type in the ontology. Any state change in that object type emits an Event to the event log, enabling process mining.

Key Insight The pipeline is the object type's source of truth. Once a pipeline successfully runs and binds to an object type, all future data comes through that pipeline — not direct connector pulls. This mirrors how Palantir Foundry works: datasets are owned by transforms, not raw sources.

2. The Nexus Development Philosophy

2.1 Event-First Thinking

In a Nexus-native app, every meaningful state change emits an event. Not as an afterthought — as a first-class design decision made before you write a single route.

Ask yourself before building any feature:

  • What is the case? The thing being tracked over time (a project, incident, order, patient).
  • What are the activities? Discrete named transitions (PROJECT_CREATED, STAGE_MOVED, COMMENT_ADDED).
  • What is the timestamp? When did this transition actually happen?
  • What attributes matter? Who did it, from what state, to what state, with what payload?

The event log answers "what happened and when" for any case. Process mining then automatically discovers bottlenecks, variants, and deviations — without any extra configuration.

Rule #1 If a row changes state in your database, emit an event. If something is created, emit an event. If something is assigned, moved, closed, or deleted — emit an event.

2.2 Everything Is an Ontological Object

Your application's domain objects are not just rows in a database. They are ontological objects — versioned, typed, linked entities in the Nexus knowledge graph.

When you design a new entity (say, a Project), think:

  • What are the canonical properties (id, name, status, created_at)?
  • What is the semantic type of each property (IDENTIFIER, STATUS, DATE, PERSON_NAME…)?
  • What other objects does it relate to (Project has_many Stages, belongs_to Company)?
  • What is the primary key used for upsert/merge (always one IDENTIFIER field)?

When these objects live in the ontology, every other app and pipeline can reference them. A project management app and an incident tracking app can both link to the same TeamMember object type.

2.3 Apps Expose APIs, Pipelines Own the Model

Your application's job is to expose clean, paginated REST APIs. The Nexus platform's job is to consume those APIs via connectors and pipelines and maintain the ontological model.

This separation is intentional:

  • Your app doesn't need to know about ontologies or pipelines.
  • The platform doesn't need to know about your business logic.
  • Multiple pipelines can consume the same app API differently — enriching, filtering, or combining with other sources.

3. Building a Nexus-Native Application

3.1 Backend Structure (FastAPI)

All Nexus services use FastAPI with async SQLAlchemy. Your app should follow the same pattern to feel native:

# app/main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from routers import projects, stages, comments
from database import init_db

app = FastAPI(title="My App")
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])

app.include_router(projects.router, prefix="/projects")
app.include_router(stages.router,   prefix="/stages")
app.include_router(comments.router, prefix="/comments")

@app.on_event("startup")
async def startup():
    await init_db()

@app.get("/health")
async def health():
    return {"status": "ok"}

3.2 Event Emission Pattern

Copy this pattern into every router that changes state. It fires asynchronously so it never slows down your response:

import asyncio
import httpx
from uuid import uuid4
from datetime import datetime, timezone

EVENT_LOG_URL = "http://event-log-service:8005"

async def emit_event(
    case_id: str,          # the primary entity being tracked
    activity: str,        # e.g. "PROJECT_CREATED", "STAGE_MOVED"
    object_type_id: str,  # the ontological object type
    object_id: str,       # the specific record id
    tenant_id: str,
    pipeline_id: str = "",
    connector_id: str = "",
    attributes: dict = None
):
    try:
        async with httpx.AsyncClient(timeout=3.0) as client:
            await client.post(f"{EVENT_LOG_URL}/events", json={
                "id": str(uuid4()),
                "case_id": case_id,
                "activity": activity,
                "timestamp": datetime.now(timezone.utc).isoformat(),
                "object_type_id": object_type_id,
                "object_id": object_id,
                "pipeline_id": pipeline_id,
                "connector_id": connector_id,
                "tenant_id": tenant_id,
                "attributes": attributes or {},
            })
    except Exception:
        pass  # event log failures must never break the app

# Usage inside a route:
@router.post("/projects/{pid}/stages")
async def create_stage(pid: str, body: StageIn, x_tenant_id: Optional[str] = Header(None)):
    tid = _tid(x_tenant_id)
    stage = await db_create_stage(pid, body)
    asyncio.create_task(emit_event(
        case_id=pid,
        activity="STAGE_CREATED",
        object_type_id="project",
        object_id=pid,
        tenant_id=tid,
        attributes={"stage_name": body.name, "stage_type": body.stage_type}
    ))
    return stage

3.3 Making Your API Connector-Ready

The Nexus Connector Service pulls data from your API to populate the ontology. For this to work well, your endpoints must follow these rules:

RuleWhyExample
Always return arrays for list endpoints The pipeline SOURCE node expects a list to iterate GET /projects → [{...}, {...}]
Include the primary ID in every record ENRICH and DEDUPE nodes use it for joins and dedup Every object has "id": "uuid"
Support pagination via query param Large datasets need batching GET /projects?page=1&limit=100
Support detail lookup by ID ENRICH node calls this per-row with a join key GET /projects?id=abc123
Support date filtering Incremental sync — only changed records GET /projects?updated_after=2024-01-01
Include timestamps on every record Pipeline SINK_EVENT needs a timestamp field "created_at", "updated_at"
Flat is better than nested (or provide both) The FLATTEN node can explode arrays, but flat is simpler Top-level fields preferred over deep nesting
Design Tip Add a GET /list and GET /detail?id=X pair for every major entity. List gives the pipeline all IDs; detail gives the ENRICH node the full record per row. This is the core Nexus data pattern.

4. Designing Ontological Objects

4.1 Choosing Your Entities

An ontological object type maps to a real-world entity that persists, evolves, and participates in relationships. Good candidates:

  • Things with an identity that survives beyond a single transaction
  • Things you want to query, filter, and join across systems
  • Things that go through states over time (good for process mining)

Bad candidates: lookup values, enums, intermediate join tables, raw API response envelopes.

4.2 Property Semantic Types

Every property has a semantic type that the platform uses for inference, PII detection, and display formatting:

Semantic TypeUse forPII?
IDENTIFIERPrimary keys, UUIDs, external IDsNo
PERSON_NAMEFull name, first name, last nameHigh
EMAILEmail addressesHigh
STATUSState machine values (active, closed, pending)No
DATEDate-only fields (start_date, due_date)Low
DATETIMEFull timestamps (created_at, updated_at)Low
CATEGORYType fields, tags, department codesNo
QUANTITYCounts, durations, numeric measuresNo
CURRENCYMonetary amountsNo
URLLinks, endpoints, resource pathsNo
TEXTFree text, descriptions, commentsMedium
BOOLEANFlags, yes/no fieldsNo

4.3 Relationships (Links)

Object types can be linked to each other. When you link two types, the Nexus graph shows the relationship and pipelines can traverse it:

Relationship TypeMeaningExample
has_manyOne parent, many childrenProject has_many Stages
belongs_toChild references parentStage belongs_to Project
has_oneOne-to-oneProject has_one ProjectManager
many_to_manyMany-to-manyProject many_to_many TeamMember

Links require join keys — the field on each side that connects the records. For example: Stage.project_id → Project.id.

5. Connecting Through Pipelines

5.1 The Standard Pipeline Pattern

SOURCE
→
FILTER
→
MAP
→
SINK OBJECT

The SOURCE node fetches all records from a connector. Optional transform nodes clean the data. SINK OBJECT pushes records to an Object Type via POST /object-types/{id}/records/ingest — which upserts them into the ontology.

5.2 ENRICH — For Multi-Endpoint APIs

SOURCE (list)
→
ENRICH (detail)
→
SINK OBJECT

When your list endpoint returns summary records and a separate detail endpoint returns the full record per ID, use ENRICH:

  • Join Key — the field on the list row that holds the ID (e.g. id)
  • Lookup Param — the query param on the detail endpoint (e.g. id)
  • Detail Connector — the connector pointing at your detail endpoint

The executor calls the detail endpoint once per row, concurrently (10 at a time), merges the result onto the row, and passes the fat record to the next node.

5.3 SINK EVENT — For Process Mining

SOURCE
→
SINK OBJECT
→
SINK EVENT

Add a SINK EVENT node after SINK OBJECT to convert records into process mining events. Configure:

  • Activity Field — which field contains the activity name (e.g. status, stage_type)
  • Case ID Field — which field is the case identifier (e.g. project_id)
  • Timestamp Field — which field is the event time (e.g. updated_at)
Important The SINK EVENT node generates one event per record, per run. For true process mining, emit events from your app's routes (Section 3.2) — those capture real-time state transitions, not batch snapshots.

4.2 The Database Pattern (JSONB Row Storage)

Every Nexus service uses the same storage pattern: a SQLAlchemy table with a data JSON column that holds the full serialized object. This avoids schema migrations when you add fields to your objects — you just put them in the JSON blob.

# database.py
import os
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession, async_sessionmaker
from sqlalchemy.orm import DeclarativeBase
from sqlalchemy import Column, String, DateTime, JSON, func

DATABASE_URL = os.environ.get(
    "DATABASE_URL",
    "postgresql+asyncpg://nexus:nexus_pass@postgres:5432/nexus"
)
engine = create_async_engine(DATABASE_URL, echo=False)
AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)

class Base(DeclarativeBase):
    pass

# One Row class per entity. Pattern: indexed columns for querying,
# data JSON column for everything else.
class ProjectRow(Base):
    __tablename__ = "my_app_projects"   # prefix with app name to avoid conflicts
    id         = Column(String, primary_key=True)
    tenant_id  = Column(String, nullable=False, index=True)
    company_id = Column(String, nullable=False, index=True)  # index FK you filter by
    data       = Column(JSON, nullable=False)                   # full object as JSON
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), server_default=func.now(), onupdate=func.now())

class StageRow(Base):
    __tablename__ = "my_app_stages"
    id         = Column(String, primary_key=True)
    tenant_id  = Column(String, nullable=False, index=True)
    project_id = Column(String, nullable=False, index=True)
    data       = Column(JSON, nullable=False)

# Tables are created automatically on startup — no migrations needed.
async def init_db():
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

async def get_session() -> AsyncSession:
    async with AsyncSessionLocal() as session:
        yield session
Critical: flag_modified When you mutate a JSON column in-place (e.g. row.data["progress"] = 80), SQLAlchemy won't detect the change automatically. You MUST call from sqlalchemy.orm import attributes; attributes.flag_modified(row, "data") before committing, or the update will be silently lost.

4.3 A Complete Router (CRUD + events)

# routers/projects.py
from fastapi import APIRouter, Depends, HTTPException, Header
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from sqlalchemy.orm import attributes
from pydantic import BaseModel
from uuid import uuid4
from datetime import datetime, timezone
from typing import Optional
import asyncio
from database import get_session, ProjectRow
from events import emit_event  # shared helper in your service

router = APIRouter()

class ProjectIn(BaseModel):
    name: str
    status: str = "active"
    company_id: str
    description: str = ""

# ── LIST (flat, connector-friendly) ──────────────────────────────
@router.get("")
async def list_projects(
    id: Optional[str] = None,          # ENRICH detail lookup: ?id=abc
    status: Optional[str] = None,       # ?status=active
    updated_after: Optional[str] = None, # incremental sync
    x_tenant_id: Optional[str] = Header(None),
    db: AsyncSession = Depends(get_session),
):
    tid = x_tenant_id or "tenant-001"
    stmt = select(ProjectRow).where(ProjectRow.tenant_id == tid)
    if id:   stmt = stmt.where(ProjectRow.id == id)
    rows = (await db.execute(stmt)).scalars().all()
    return [r.data for r in rows]  # returns array — connector-ready

# ── CREATE ────────────────────────────────────────────────────────
@router.post("", status_code=201)
async def create_project(
    body: ProjectIn,
    x_tenant_id: Optional[str] = Header(None),
    db: AsyncSession = Depends(get_session),
):
    tid = x_tenant_id or "tenant-001"
    pid = str(uuid4())
    now = datetime.now(timezone.utc).isoformat()
    data = {"id": pid, "created_at": now, "updated_at": now, **body.model_dump()}
    db.add(ProjectRow(id=pid, tenant_id=tid, company_id=body.company_id, data=data))
    await db.commit()
    asyncio.create_task(emit_event(
        case_id=pid, activity="PROJECT_CREATED",
        object_type_id="project", object_id=pid, tenant_id=tid,
        attributes={"name": body.name, "company_id": body.company_id}
    ))
    return data

# ── UPDATE ────────────────────────────────────────────────────────
@router.put("/{pid}")
async def update_project(
    pid: str, body: ProjectIn,
    x_tenant_id: Optional[str] = Header(None),
    db: AsyncSession = Depends(get_session),
):
    tid = x_tenant_id or "tenant-001"
    row = (await db.execute(
        select(ProjectRow).where(ProjectRow.id == pid, ProjectRow.tenant_id == tid)
    )).scalar_one_or_none()
    if not row:
        raise HTTPException(404)

    old_status = row.data.get("status")  # capture before overwriting
    row.data = {**row.data, **body.model_dump(),
                "updated_at": datetime.now(timezone.utc).isoformat()}
    attributes.flag_modified(row, "data")  # REQUIRED for JSON mutation
    await db.commit()

    if old_status != body.status:
        asyncio.create_task(emit_event(
            case_id=pid, activity="PROJECT_STATUS_CHANGED",
            object_type_id="project", object_id=pid, tenant_id=tid,
            attributes={"from": old_status, "to": body.status}
        ))
    return row.data

4.6 Registering in docker-compose.yml

Add your service to docker-compose.yml following this pattern. Pick the next available port (currently 8008 is the last — use 9000+):

# docker-compose.yml — add this block
my-app-service:
  build:
    context: ./backend               # build from backend/ so shared/ is available
    dockerfile: my_app_service/Dockerfile
  ports:
    - "9000:9000"
  environment:
    - DATABASE_URL=postgresql+asyncpg://nexus:nexus_pass@postgres:5432/nexus
    - EVENT_LOG_SERVICE_URL=http://event-log-service:8005
    # Add any other service URLs you need to call:
    - ONTOLOGY_SERVICE_URL=http://ontology-service:8004
  depends_on:
    postgres:
      condition: service_healthy
  networks:
    - nexus-net
  command: uvicorn main:app --host 0.0.0.0 --port 9000 --reload
Network All services communicate using their Docker service name as the hostname (e.g. http://event-log-service:8005). This only works when both services are on the same nexus-net network. Never use localhost for inter-service calls inside Docker.

4.7 Inter-Service Communication

Use httpx.AsyncClient for all calls to other Nexus services. Read the target URL from an environment variable so it works both in Docker and local development:

import os
import httpx

ONTOLOGY_URL = os.environ.get("ONTOLOGY_SERVICE_URL", "http://ontology-service:8004")

# Calling another service — always use async with and a timeout
async with httpx.AsyncClient(timeout=10.0) as client:
    resp = await client.get(
        f"{ONTOLOGY_URL}/object-types",
        headers={"x-tenant-id": tid}   # always forward tenant_id
    )
    resp.raise_for_status()
    data = resp.json()

Implementing differential event emission (update routes)

@router.put("/{pid}")
async def update_project(pid: str, body: ProjectIn, x_tenant_id: Optional[str] = Header(None)):
    tid = _tid(x_tenant_id)
    old = await db_get_project(pid)  # read before updating
    updated = await db_update_project(pid, body)

    # Emit targeted event based on what changed
    if old["status"] != body.status:
        asyncio.create_task(emit_event(
            case_id=pid, activity="PROJECT_STATUS_CHANGED",
            object_type_id="project", object_id=pid, tenant_id=tid,
            attributes={"from": old["status"], "to": body.status}
        ))
    if old["pmId"] != body.pm_id:
        asyncio.create_task(emit_event(
            case_id=pid, activity="PROJECT_PM_ASSIGNED",
            object_type_id="project", object_id=pid, tenant_id=tid,
            attributes={"from_pm": old["pmId"], "to_pm": body.pm_id}
        ))
    return updated

7. Claude Prompt for Nexus-Native Development

Paste this prompt at the beginning of any Claude session when building Nexus apps:

You are building a Nexus Origins-native application. Follow these rules:

## PLATFORM CONTEXT
Nexus Origins is an enterprise data integration platform with 8 microservices:
- Connector Service (:8001) — API credential management and schema discovery
- Pipeline Service (:8002) — DAG executor (SOURCE → transforms → SINK) with built-in scheduler
- Inference Service (:8003) — Claude AI for semantic type detection
- Ontology Service (:8004) — Versioned object types, records, links, and app registry
- Event Log Service (:8005) — Process mining event store (case/activity/timestamp)
- Audit Service (:8006) — Immutable audit trail
- Schema Registry (:8007) — Raw schema versioning
- Correlation Engine (:8008) — Cross-source entity resolution

## RULE 1: EVENT-FIRST DESIGN
Before writing any route, define:
1. case_id — the primary entity being tracked over time (project_id, incident_id, order_id)
2. activities — named state transitions (PROJECT_CREATED, STAGE_MOVED, COMMENT_ADDED)
3. timestamps — when each transition happened
4. attributes — who did it, from what state, to what state

Every meaningful state change MUST emit an event asynchronously to the Event Log Service.
Event log failures must never block the response (always fire-and-forget with try/except).

## RULE 2: CONNECTOR-READY APIs
Every entity needs:
- A flat list endpoint: GET /entities (returns array, supports ?updated_after=ISO for incremental sync)
- A detail lookup: GET /entities?id={id} (for ENRICH node per-row calls)
- Pagination: ?page=N&limit=100
- Always include: id, created_at, updated_at on every record

## RULE 3: ONTOLOGICAL THINKING
For every entity, define:
- Primary key field (semantic type: IDENTIFIER)
- Status field if it has states (semantic type: STATUS)
- All timestamp fields (semantic type: DATETIME or DATE)
- PII fields (PERSON_NAME, EMAIL → pii_level: HIGH)
- Foreign keys to related objects (semantic type: IDENTIFIER)

## RULE 4: TENANT ISOLATION
Every route reads x-tenant-id from header:
  def _tid(x_tenant_id: Optional[str]) -> str:
      return x_tenant_id or "tenant-001"
All DB queries must be scoped: WHERE tenant_id = tid

## RULE 5: ASYNC SQLALCHEMY PATTERN
Always use async SQLAlchemy with AsyncSession.
Use flag_modified(row, "data") when mutating JSONB fields.
Use uuid4() for all IDs. Store ISO timestamps (datetime.now(timezone.utc).isoformat()).

## RULE 6: PIPELINE DESIGN
When designing pipelines for your app:
- SOURCE → SINK_OBJECT for simple entities
- SOURCE → ENRICH → SINK_OBJECT when list + detail endpoints exist
- Add SINK_EVENT after SINK_OBJECT for entities that participate in process mining
- Set pollFrequency on SOURCE: 5m for real-time, 1h for hourly, 1d for daily

## EVENT EMISSION TEMPLATE
async def emit_event(case_id, activity, object_type_id, object_id, tenant_id, attributes=None):
    try:
        async with httpx.AsyncClient(timeout=3.0) as client:
            await client.post("http://event-log-service:8005/events", json={
                "id": str(uuid4()),
                "case_id": case_id,
                "activity": activity,
                "timestamp": datetime.now(timezone.utc).isoformat(),
                "object_type_id": object_type_id,
                "object_id": object_id,
                "pipeline_id": "",
                "connector_id": "",
                "tenant_id": tenant_id,
                "attributes": attributes or {},
            })
    except Exception:
        pass  # never block on event log

# In every state-changing route:
asyncio.create_task(emit_event(...))

When given an application to build, produce in order:
1. Entity list with semantic types and PII classification
2. Event emission plan (case_id, all activities, key attributes)

9.3 The Zustand Store Pattern

All frontend state lives in Zustand stores. Each store owns one domain, fetches from the backend, and exposes typed actions. Components never call fetch() directly — they call store methods.

// store/myStore.ts
import { create } from 'zustand';

const API = import.meta.env.VITE_MY_SERVICE_URL || 'http://localhost:9000';

interface MyStoreState {
  items: MyItem[];
  loading: boolean;
  fetchItems: () => Promise<void>;
  addItem: (item: MyItem) => Promise<MyItem>;
}

export const useMyStore = create<MyStoreState>((set) => ({
  items: [], loading: false,

  fetchItems: async () => {
    set({ loading: true });
    const data = await fetch(`${API}/items`).then(r => r.json());
    set({ items: data, loading: false });
  },

  addItem: async (item) => {
    const created = await fetch(`${API}/items`, {
      method: 'POST', headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(item),
    }).then(r => r.json());
    set((s) => ({ items: [...s.items, created] }));
    return created;
  },
}));

// In your component:
const { items, loading, fetchItems } = useMyStore();
useEffect(() => { fetchItems(); }, []);

9.8 Shared Models

If your entity is referenced by multiple services (pipelines, ontology, etc.), add it to backend/shared/models.py. It is copied into every service image at build time. Never redefine a model locally if it already exists in shared — local copies silently diverge.
