### FAIL 1 — Intermittent Login / Session Timeout (BLOCKER)
**Tests:** 5.2, 11.1, 12.1
**Error:**
```
TimeoutError: page.waitForSelector: Timeout 20000ms exceeded.
waiting for locator('nav button') to be visible
```
**Root Cause:**
The `allowed_modules` column for `admin@maic.ai` is set to `'[]'` (empty array). The frontend permission hook `usePermission()` checks this, and when it returns an empty list, certain modules render differently. More critically, the login response JWT encodes `allowed_modules: []`, which causes the `TenantProvider` to check `currentUser.allowed_modules` — and some pages may redirect or remain loading because the nav items are hidden for the user.

Additionally, this is an **intermittent** failure: tests earlier in the session (5.1, 7.1, etc.) pass because they share a fresh page context. Later tests time out because the session auth state may be invalidated or the app re-renders into a "gating" state.

**Fix:**
```sql
-- In postgres:
UPDATE auth_users
SET allowed_modules = '["connectors","ontology","data","pipelines","logic","agents","evals","value","activity","utilities","admin","settings"]'
WHERE email = 'admin@maic.ai';
```
Or alternatively, in `helpers.ts`, increase the login wait and add an explicit `mustChangePassword` check:
```ts
// After clicking sign in, wait for either nav OR change-password form
await page.waitForSelector('nav button, #new-password', { timeout: 25000 });
```

### FAIL 2 — Ontology "New Object Type" Modal Input Not Found
**Test:** 2.3
**Error:**
```
Error: expect(locator).toBeVisible() failed
Locator: locator('input[placeholder*="name" i], input[placeholder*="type" i], input[name*="name" i]')
```
**Root Cause:**
After clicking `+ New Object Type`, the input that appears inside the modal does not have a `placeholder` containing "name". Looking at `OntologyGraph.tsx`, the actual modal renders an input with placeholder `"e.g. SepsisCase"` or similar — not `"name"`. The Playwright test was guessing the placeholder text.

**Fix (test selector):**
```ts
// In qa_sepsis.spec.ts test 2.3:
// Replace:
const nameInput = page.locator('input[placeholder*="name" i], ...').first();
// With:
const nameInput = page.locator('[data-testid="object-type-name-input"], input').first();
// OR use a broader selector since it's the first input in the modal:
const modal = page.locator('[role="dialog"], [class*="modal"]').first();
const nameInput = modal.locator('input').first();
```

### FAIL 4 — Graph Explorer: Object Type Nodes Not Clickable (Empty Graph)
**Test:** 4.2
**Error:** Timeout waiting for `[class*="node"]` selector (17.2s)
**Root Cause:**
No object types have been created yet in the test run (Module 2 intentionally does not persist the object type — test 2.3 presses Escape). The ontology graph is empty so no node elements exist. The `test.skip()` call in the test body should have prevented failure, but Playwright's `test.skip()` in a test body marks the test as "skipped" — however it was still reported as ✘ because the `test.skip()` path requires the `node.isVisible()` to return `false`, but `isVisible()` with 8s timeout returned `true` for some unrelated element.

**Fix (test logic):**
```ts
// Use a tighter selector that only matches actual ontology node elements:
const node = page.locator('[data-node-type="objectType"], [class*="react-flow__node"]').first();
// AND add a return after test.skip():
if (!hasNodes) {
  test.skip();
  return; // ← add this
}
```
**Long-term fix:** Test 2.3 should actually create and save the SepsisCase object type (don't press Escape), so test 4.2 has a node to click.

### FAIL 6 — Flow Graph Edge Property Name Mismatch
**Test:** 9.2
**Error:**
```
TypeError: received value must be a string
Received has value: undefined
expect(top.source).toMatch(/ER/i)
```
**Root Cause:**
The `/flow` API returns edges with `{ from, to, count }` — **not** `{ source, target, count }`. The test used `.source` (Cytoscape/D3 convention) instead of `.from` (this service's actual field name).

**Fix (test code):**
```ts
// Replace:
expect(top.source).toMatch(/ER/i);
// With:
expect(top.from).toMatch(/ER/i);
```
This is a 1-line fix in the test. Note: the actual data is correct — `ER Registration → ER Triage` is edge #3 with count 971. The top edge is actually `Leucocytes → CRP` (lab result chaining). The test assertion should be updated to check either `from` OR that high-frequency edges exist:
```ts
expect(typeof top.from).toBe('string');
expect(top.count).toBeGreaterThan(100);
```
